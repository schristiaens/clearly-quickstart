AGENTS.md — Secure Coding Rules for AI Agents      Project Overview This system orchestrates four concurrent AI coding agents, each in an isolated git worktree, to collaboratively generate a technical HTML presentation. Human supervisors coordinate agent activity and enforce strict file ownership and contract-based workflows. Security sensitivity arises from handling API credentials (ElevenLabs), enforcing agent isolation, and preventing cross-agent interference or credential leaks.    Absolute Rules (NEVER do these)    NEVER hardcode credentials, API keys, or tokens in any source file or configuration; ALWAYS use environment variables injected at runtime (ELEVENLABS_API_KEY, ELEVENLABS_VOICE_ID).    NEVER commit sensitive audio files (e.g., rob-voice-sample.mp3) or any voice clone artifacts to the repository; ensure .gitignore covers these files.    NEVER modify files outside your assigned ownership zone; each agent must only write to its designated directory as defined in the File Ownership Map.    NEVER use non-system fonts or load external fonts in CSS; use only system monospace or retro-digital fonts.    NEVER use the Web Speech API for narration except as a fallback when an audio file is missing.    NEVER include external src= or href= references in the assembled HTML except for CDN-hosted assets; all other assets must be inlined.    NEVER expose the ElevenLabs API key or voice ID in logs, error messages, or client-side code.    NEVER introduce new external service integrations or dependencies without explicit security review and approval from a human supervisor.    NEVER bypass or disable file ownership enforcement, agent isolation, or worktree boundaries.    NEVER add user input handling, HTTP endpoints, or database access without explicit design and security review.    Required Patterns (ALWAYS do these)    ALWAYS store the ElevenLabs API key and voice ID as environment variables (ELEVENLABS_API_KEY, ELEVENLABS_VOICE_ID) set at runtime and never committed to the repository.    ALWAYS validate that all generated HTML, CSS, and JS are self-contained and do not reference non-CDN external resources.    ALWAYS run the assembly script to inline all assets and validate the final presentation before marking a task complete.    ALWAYS enforce file ownership boundaries by operating only within your assigned directory and using isolated git worktrees.    ALWAYS document any new agent workflow or file ownership change in the File Ownership Map and obtain supervisor approval.    ALWAYS use the ElevenLabs API for narration audio generation; fallback to Web Speech API only if the audio file is missing for a slide.    ALWAYS ensure that any environment variable required for agent operation is documented in the project README and not in code.    Data Handling Rules      Credentials (API keys, tokens):      MUST be stored only as environment variables set at runtime (ELEVENLABS_API_KEY, ELEVENLABS_VOICE_ID).    MUST NOT be committed to the repository, included in logs, or exposed in client-side code.    MUST NOT be shared between agents except via documented environment variables.    No PII, financial, health, or other sensitive personal data is processed; if this changes, update this section and obtain security review.      Security Boundaries The trust model enforces strict isolation between agents via file ownership and isolated git worktrees. The only untrusted input enters during pre-work, where a human provides audio or video files or a YouTube URL for voice cloning. This input is processed by a human and uploaded to ElevenLabs using an API key stored as an environment variable. All other data flows are between agents via files in exclusive directories.      Entry points for untrusted input:      Human-provided audio/video files or URLs for voice cloning (pre-work phase)    ElevenLabs API responses (MP3 audio files)    Before writing code that touches these entry points, AI agents MUST:      Ensure no direct user input is processed by agents; all pre-work is human-mediated.    Validate that API responses are saved only in assigned directories and not exposed to unauthorized agents.    Operations requiring explicit security review:      Any new integration with external services or APIs    Any change to file ownership boundaries or agent isolation mechanisms    Any addition of user input handling or network-exposed features      Secrets & Credentials      Secrets for local development are set as environment variables at runtime; there is no .env file or secrets manager in use.    Required environment variables:      ELEVENLABS_API_KEY    ELEVENLABS_VOICE_ID    Secrets scanning:      gitleaks is used for automated secret scanning; to run locally, execute: gitleaks detect --source .    No pre-commit hooks or automated enforcement are currently configured; manual vigilance is required.      Authentication & Authorization      No user authentication mechanism is implemented; the system is agent-driven and not user-facing.    Authorization is enforced via ownership-based file access: each agent may only modify files in its assigned directory as per the File Ownership Map.    AI agents adding new endpoints or features that could expose user input or require authentication MUST obtain explicit design and security review before implementation.      Dependency & Supply Chain Rules      To add new dependencies, obtain explicit approval from a human supervisor and document the change in the project README.    No dependency scanner is currently in use; if dependencies are added, integrate a tool such as Dependabot or Snyk and update this section.    Banned/restricted packages:      Any package or library that enables external network access, user input handling, or cryptographic operations without review.      CI/CD Security Gates      No automated security checks are currently configured in the CI/CD pipeline; all security validation is manual.    Before merging, manually run gitleaks for secret scanning and validate that all assets are inlined and no forbidden patterns are present.    If a security gate fails (e.g., secret detected), halt the merge and remediate before proceeding.      Security Gaps & Known Risks      Known Risk: No automated secret scanning or pre-commit hooks are enforced; do not add code that relies on manual secret management without review.    Known Risk: No automated dependency or static analysis scanning is in place; do not introduce new dependencies without explicit approval and security review.    Known Risk: No HTTP security headers or cryptographic standards are configured; do not add network-exposed features or user input handling without explicit security design and review.    Known Risk: The deployment environment is unspecified; do not assume any server-side or cloud security controls are present.    Known Risk: No formal authentication or MFA is implemented; do not add privileged operations or user-facing features without explicit security controls.
